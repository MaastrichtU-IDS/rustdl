# rustdl 0.1 Release — Design

**Status:** spec, awaiting plan
**Date:** 2026-06-04
**Context:** Today's session ended a sustained push (`outperform-hermit-plan.md` is now empirically met; `perf-2026-06-04-konclude-vs-rustdl.md` shows rustdl beats HermiT on every measured workload, wins outright vs Konclude on Horn workloads, and is at 3.1× Konclude on ORE-10908). The next perf step is Konclude-style global classification — a 6-10 week multi-month rewrite. Before committing to that, ship 0.1 as a real release to (a) surface what users actually need vs perf chase, (b) make the work visible / usable, (c) anchor a stable v0.x→v1 progression.

## Goal

Ship rustdl 0.1.0 — a release that's discoverable, installable, and accurately documented. The engine work is already done; this is a documentation + packaging + release-mechanics project.

## Non-goals

- New engine features.
- Konclude-style global classification (deferred — multi-month, recon-required first).
- Publishing binary distributions / package-manager formulae.
- Tutorial / cookbook content (defer to 0.1.1+).

## Scope

Five must-haves; nothing else in this release.

### M2 — Workspace version bump `0.0.1 → 0.1.0`

Root `Cargo.toml` has `[workspace.package] version = "0.0.1"`. All seven publishable crates inherit it via `version.workspace = true`. Single edit: change the workspace version line.

### M3 — Create `CHANGELOG.md`

A one-pager. Format: Keep-a-Changelog. Sections for `0.1.0` only:

- **Added:** Sound SROIQ classifier with hybrid saturation+tableau orchestrator; hypertableau wedge accelerator (default); per-class label heuristic (Phase 7); cache-deadline decoupling (Phase 8); Horn-shortcircuit fast path (Phase 2b); ABox consistency check (Phase A1, 7 sound clash patterns); datatype preprocessing (D1-D5); 9-corpus closure-diff regression harness.
- **Performance:** GALEN 445s → 0.49s (now beats Konclude); notgalen 1168s → 0.78s; ORE-10908 17× → 3.1× Konclude; sio-stripped 4.3× absolute improvement.
- **Known limitations:** Data-axiom patterns outside D4/D5 silently dropped (sound under-approximation); HasKey not supported (errors); SWRL silently skipped; role chains length >2 error; family-class ABox-saturation gap (dead-end §21); ore-15672 3-class intrinsic intractability (dead-end §18).
- **Dead-ends documented:** 21 entries in `docs/hypertableau-dead-ends.md`.

Pre-0.1 entries optional; the changelog can start from 0.1.0 as "first tagged release".

### M4 — `cargo publish --dry-run` for each publishable crate

Run `cargo publish --dry-run` from each of the seven publishable crates (`owl-dl-core`, `owl-dl-saturation`, `owl-dl-tableau`, `owl-dl-datatypes`, `owl-dl-reasoner`, `owl-dl-cli`, `owl-dl-bench`). Catches:
- Missing `description` / `repository` / `keywords` / `categories`.
- Dependency-version constraints incompatible with publication (path deps without version).
- Files that should be excluded but aren't (`target/` / large fixtures).

Fix anything that surfaces. Do NOT actually publish to crates.io in this release — confirm dry-run clean, then a follow-up commit can publish if/when we decide to.

### M5 — Verify the repository URL

Root `Cargo.toml` references `repository = "https://github.com/dumontierlab/rustdl"` and `homepage = "https://github.com/dumontierlab/rustdl"`. Verify:
- The GitHub org `dumontierlab` exists and the user has push access.
- The repo `rustdl` exists OR plan to create it before publishing.
- If the URL is wrong (or the repo doesn't exist yet), either update the URL to wherever rustdl actually lives, or note that the URL is aspirational + add a `mkdir / git remote add` task to set it up.

If repo creation / URL change is needed, that's a separate commit (not bundled with the README refresh).

## Architecture (such as it is)

Just edits. No new modules, no new tests, no new features.

```
edit:   README.md                          # M1
edit:   Cargo.toml (workspace.package)     # M2
create: CHANGELOG.md                       # M3
run:    cargo publish --dry-run × 7        # M4
verify: github.com/dumontierlab/rustdl     # M5
```

## Error handling

If M4's dry-runs surface metadata gaps (most common: a crate missing `categories` or having a path-only dep), fix inline. If a fix is non-trivial (e.g., a dep cycle that needs restructuring), escalate as a separate scoping question rather than blocking the release.

If M5 reveals the repo doesn't exist at the claimed URL, surface to the user — repo-creation is out-of-scope for an LLM agent.

## Testing

Tests are already in place from the engine work (78 lib tests + 18 ABox consistency tests + 9 corpus closure-diffs + many doctests). The release adds none. Pre-commit verification:
- `cargo build --workspace --release` succeeds
- `cargo test --workspace` shows no NEW failures vs main baseline (6 pre-existing lib test failures are acceptable per session-arc context)
- `cargo clippy --workspace --all-targets --all-features -- -D warnings` clean

## Performance contract

None. Documentation work doesn't affect engine wall.

## Out-of-scope items captured for 0.1.1+

- Wiring `owl-dl-datatypes` for true concrete-domain reasoning (D5 covers the preprocessing patterns we have).
- ABox saturation pass (dead-end §21 follow-on).
- Documentation site (docs.rs polish, mdbook, etc.).
- Logo / branding.
- Tutorial / cookbook.
- Binary distributions / package-manager formulae.
- Performance regression CI gate.

## Release sequencing

After this spec is approved, the implementation plan (separate writing-plans invocation) breaks M1-M5 into bite-sized tasks. The natural order:
1. M5 first (verify repo URL — if wrong, all other work needs the right URL).
2. M2 (version bump — single-line edit).
3. M1 (README refresh — biggest single deliverable).
4. M3 (CHANGELOG).
5. M4 (publish dry-runs — last because metadata fixes from M4 may touch README again).
6. Final commit + git tag `v0.1.0`.

No remote-push or crates.io-publish is part of the release sequence; both require explicit user authorization per project convention.
